Design Issues:

Many issues were evaluated when designing Shortqut. With respect to reusability, many parts of the program would be useful in other programs, especially those dealing with GPS units, distance calculations and pathfinding. Our code will also adhere to strict style guidelines that are standards in the python community, thus enhancing maintainability. The style of development that we chose, iterative in the REPL, means that we are testing while developing. Tests will be run on atomic chunks of code while writing them. Unit tests will also be used when possible and functions and functional units will be tested using unit tests designed for them. Python is not known for speed. It is an interpreted language. However, we chose it for it's easy syntax and vast amount of libraries. Any performance bottlenecks that are identified on our target platform (Atom based netbooks) will be refactored or written in C to speed them up. Most sections of code will not be computationally bound but rather waiting on I/O. Python was also chosen for it's good portability. We are currently targetting Windows and Linux but the Android platform would not be impossible to port to. Python prototyping is quick and easy because there is no compilation time and code can be run directly in the interpreter. There are several difficult problems which may be hard to implement. The algorithm to decide how long a road takes to traverse and the algorithm to find the best 3 routes will be hard. The only major drawback to python other than its performance is that only one person on the team knows it. Fortuneately python is easy to learn.
